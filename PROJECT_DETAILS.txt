RoadSoS — Project Details, Packages, and Assumptions
================================================================================
IIT Madras Road Safety Hackathon | Last updated: May 2026

This file summarizes software used across the monorepo (backend, web frontend,
Android app), external services, and design assumptions. For setup steps see
README.md; for Android-only setup see app/README.md.

--------------------------------------------------------------------------------
REPOSITORY LAYOUT
--------------------------------------------------------------------------------
  backend/     FastAPI API, PostGIS data, Groq intent, OSM Overpass fetch
  frontend/    React + Vite web app (PWA)
  app/         Kotlin + Jetpack Compose Android app (git branch: app)
  docker-compose.yml   PostgreSQL 15 + PostGIS container

Active development branches (typical):
  main       — stable baseline
  ui-polish  — web emergency features (accident mode, share link, SOS card)
  app        — native Android client (does not modify main by convention)

--------------------------------------------------------------------------------
RUNTIME / TOOLING REQUIREMENTS
--------------------------------------------------------------------------------
  Python 3.10+ (backend)
  Node.js 18+ and npm (frontend)
  Docker Desktop (PostGIS database)
  JDK 17 or Android Studio bundled JBR 21 (Android builds; Java 25 is not
    supported by current Gradle/Kotlin tooling)
  Android Studio Ladybug 2024.2+ (recommended for app/)
  Android SDK 35, min device API 26 (Android 8.0)

--------------------------------------------------------------------------------
API ENDPOINTS (shared by web and Android)
--------------------------------------------------------------------------------
  GET  /health
  POST /chat/                    { message, user_lat?, user_lng? }
  GET  /geocode/?q=               India place names
  GET  /emergency/country?lat=&lng=
  POST /emergency/nearby           lat, lng, radius_km, types[], accident_mode,
                                   per_type_limit

  Default dev URLs:
    Backend:  http://localhost:8000  (host) | http://10.0.2.2:8000 (emulator)
    Frontend: http://localhost:5173  | http://10.0.2.2:5173 (emulator)
    Physical Android device: http://YOUR_LAN_IP:8000, with backend started on
      host 0.0.0.0 rather than 127.0.0.1.

--------------------------------------------------------------------------------
ASSUMPTIONS
--------------------------------------------------------------------------------

Geography & data
  - Primary demo region is India; forward geocoding uses countrycodes=in.
  - Reverse geocoding and country emergency numbers support multiple countries
    (IN, US, GB, AU, CA, default 112).
  - Hospital results filter out specialty clinics (dental, ayurveda, etc.) via
    name keywords and OSM tags where available.
  - Service data is seeded from Overpass on demand; PostGIS caches by osm_id.
  - Towing services map to OSM type "towing" (car repair / tyre shops).

AI / chat
  - Groq API key is required for real intent parsing; without it backend falls
    back to keyword/heuristic behavior (e.g. accident regex).
  - Accident phrases ("I just had an accident", crash, collision) trigger
    accident_mode with all four types: hospital, ambulance, police, towing.
  - per_type_limit defaults to 2 per type in accident mode (max ~8 markers).

Clients
  - Web and Android are thin clients; all business logic for search/filtering
    lives on the backend except UI grouping and offline SOS card cache.
  - CORS is wide open (*) in development for the React app only; native Android
    does not use CORS.
  - Share links assume the web app is deployed with SPA fallback for /emergency.
  - Android uses ACTION_DIAL (no CALL_PHONE permission) for emergency calls.
  - Cleartext HTTP to local backend is allowed in Android debug builds only.
  - Physical device testing requires backend on 0.0.0.0 and LAN IP in API_BASE_URL.
  - Android chat controls live in the Scaffold bottomBar so the floating SOS
    action stays above the input/send row and does not cover it.

Security & production
  - config.py, .env, and local.properties are not committed.
  - Production API URL, Maps keys, and HTTPS are left to deployer configuration.
  - No end-user authentication; app is for emergency lookup, not PII storage.

Offline / PWA
  - Web service worker caches /emergency/nearby responses; chat is not cached.
  - Android SOS card uses DataStore after at least one successful online search.
  - Offline lists may be stale; national numbers in guidance are static copy.

Maps
  - Web uses Leaflet + CartoDB; Android requires a valid Google Maps SDK key.
  - Without MAPS_API_KEY the Android app still runs but map tiles may not load.

Git workflow
  - Android work targets branch "app"; main branch is not modified for app features.
  - Monorepo: open Android Studio on the app/ directory, not repo root.
